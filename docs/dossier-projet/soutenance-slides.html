Contexte métier

Collector : d'organisateur de salons à marketplace

La start-up

  • 5 ans d'existence, orientée événementiel (salons d'objets de collection) avant la levée de fonds par crowdfunding.
  • Nouvelle équipe IT : 1 Lead Developer (ce rôle) + 2 développeur·euse·s confirmé·e·s (5 ans d'XP).
  • SI existant limité : Office 365, site vitrine WordPress — aucune base technique produit à reprendre.

Pourquoi la sécurité est prioritaire

  • L'application porte des transactions financières réelles — commission de 5 % prélevée par la plateforme.
  • Paiement exclusivement via la plateforme : les échanges directs acheteur/vendeur (email, téléphone) doivent être techniquement empêchés.
  • Contrainte explicite du contexte : « l'application incluant des transactions financières, la sécurité est une exigence de premier plan ».

Ma mission (Lead Developer)

Définir un processus de développement conforme aux exigences de qualité logicielle (référentiel ISO 25010), livrer une V1 avec pipeline CI/CD testé, déploiement orchestré et sécurité minimale, puis auditer cette V1 pour proposer un plan de remédiation priorisé.

Cadrage fonctionnel

Exigences clés de collector.shop

Utilisateurs & rôles

  • 2 profils : acheteur / vendeur (cumulables), espace dédié avec suivi achats/ventes, notation, historique.
  • Admin : back-office, catégories, modération articles/vendeurs.
  • Chat intégré acheteur↔vendeur, modérable par l'admin.
  • Inscription obligatoire pour vendre/acheter ; catalogue navigable sans authentification.

Transaction & contrôle

  • Paiement CB uniquement, via la plateforme — jamais en direct.
  • Échange de coordonnées personnelles interdit dans la messagerie.
  • Publication d'article soumise à contrôle avant mise en vente, automatisable au maximum.
  • Variation de prix collectée, notifiée aux intéressés + à un composant back-office anti-fraude.

Évolutivité attendue

  • Recommandations par centre d'intérêt (V1), puis parcours catalogue (V2).
  • Intégration future d'un outil de détection de fraude (interne ou tiers).
  • Internationalisation et accessibilité.
  • Architecture prête pour enchères, événements live, bot avant-vente, analytics direction.

Pourquoi ISO 25010

Chaque indicateur suivi est relié à une caractéristique de qualité (fiabilité, sécurité, maintenabilité, efficacité de performance). Le suivi continu de ces métriques dans le pipeline CI, plutôt qu'un contrôle ponctuel, est ce qui empêche l'accumulation silencieuse de dette technique : une régression est bloquée au commit, pas découverte en production.

Phase 2 · Développement & déploiement Grille : 4 pts

Architecture système — microservices event-driven

Monorepo : 1 frontend, 4 microservices Go, 3 bases PostgreSQL isolées par domaine, RabbitMQ pour le découplage événementiel.

Principes structurants

  • Découpage par domaine : identité / catalogue-vente / suivi prix / notifications — chaque service a sa propre base.
  • REST synchrone pour les opérations utilisateur, RabbitMQ pour les réactions métier asynchrones (découple catalog-service du reste).
  • Mapping déterministe uint → UUID pour exposer les IDs catalogue aux services consommateurs.
  • Communication interservice authentifiée : endpoint interne auth-service protégé par secret partagé X-Internal-Secret.

Ce que le POC valide

Le découpage applicatif, la communication interservice (REST + événements), la sécurité (JWT, secrets internes) et l'orchestration (Kubernetes/GitOps) sont tous implémentés et documentés — pas seulement schématisés sur papier.

Phase 2 · Fonctionnalité métier POC Grille : 1 pt

Achat avec validation vendeur

Le vendeur garde la main avant que la transaction ne soit actée — répond à l'exigence « paiement via la plateforme uniquement ».

User story

« En tant que vendeur, je veux être notifié et pouvoir accepter ou refuser chaque demande d'achat, afin de garder le contrôle sur mes ventes avant que la transaction ne soit finalisée. »

Critères d'acceptation testés

  • Achat propre annonce refusé (400) — TestAcceptance_CannotBuyOwnArticle
  • Achat → commande pending + article réservé — TestAcceptance_BuyCreatesPendingOrderAndReservesArticle
  • Seul le vendeur peut décider (403 sinon) — TestAcceptance_OnlySellerCanAcceptOrder
  • Refus → article redevient disponible — TestAcceptance_SellerRejectsOrderReleasesArticle
  • Décision non rejouable (409) — TestAcceptance_CannotDecideOrderTwice

Tests d'acceptation HTTP de bout en bout sur le vrai routeur Gin (middleware JWT réel inclus) — le second type de test exigé, en plus des tests unitaires.

Phase 2 · Architecture événementielle

Flux mise à jour de prix & détection de fraude

catalog-service ne connaît ni price-tracker ni notification — chaque service se teste, se déploie et scale indépendamment.

Pourquoi RabbitMQ ici

  • Une mise à jour de prix doit simultanément : s'historiser, être analysée pour fraude, et notifier les acheteurs intéressés — sans coupler ces responsabilités.
  • Règles de détection déjà implémentées dans price-tracker-service : SUSPICIOUS_SPIKE, FLOOD_PRICING, DUMPING.
  • Répond directement à l'exigence du contexte : « la notification de changement de prix doit aussi être reçue par un composant back-office de détection des fraudes ».
  • Alertes exposées via GET /api/v1/alerts, résolution admin via PUT /alerts/:id/resolve.

Phase 1 · Cycle de vie & sécurité du développement Grille : 3 pts

Pipeline CI/CD — chaîne de livraison continue GitOps

Build (SAST/SCA)

  • go vet, ESLint, Prettier
  • go test -race, vitest
  • gosec, Semgrep, SonarCloud
  • govulncheck, npm audit, Dependabot

Package

  • hadolint (Dockerfile)
  • Image distroless non-root
  • Trivy scan bloquant
  • SBOM Syft + signature cosign keyless

Release → Deploy

  • Commit du digest dans l'overlay staging (GitOps)
  • Argo CD sync auto (staging), sync manuel (prod)
  • Admission Kyverno : signature vérifiée, non-root, no-latest, resources — Enforce

Principe GitOps : la CI ne pousse jamais directement vers le cluster — elle modifie l'état désiré dans Git, Argo CD applique cet état. La production reste protégée par une promotion explicite (PR + synchronisation manuelle).

Sécurité — bonnes pratiques et remédiation

  • Endpoint interservice protégé par secret partagé
  • Chaîne d'image de confiance : distroless, Trivy, cosign, SBOM, vérification Kyverno
  • Transaction SQL anti double-vente

Phase 2 · Bac à sable Grille : 2 pts

Protocole d'expérimentation — HTTPS via cert-manager

Technologie testée & environnement

  • cert-manager + Ingress NGINX + autorité Let's Encrypt sur cluster k3s (VPS OVH, Ubuntu, 4 vCore/8 Go)
  • Piloté en déclaratif par Argo CD (App-of-Apps)
  • ClusterIssuer Let's Encrypt + ressource Certificate par overlay (staging/prod)

Étapes clés

  • Bootstrap cluster + Argo CD → déploiement cert-manager (Helm Jetstack)
  • ClusterIssuer ACME HTTP-01 → Certificate référencé par l'Ingress TLS
  • Vérification kubectl get certificate → READY=True, puis curl -v https://...

Difficultés rencontrées

  • Résolution du challenge HTTP-01 : ordre de déploiement critique (Ingress TLS actif avant émission = blocage)
  • Propagation DNS avant relance de l'émission (rate-limit Let's Encrypt à surveiller)
  • Double environnement : secrets Certificate distincts pour éviter les écrasements croisés Argo CD

Résultat & décision

HTTPS opérationnel de bout en bout, renouvellement automatique (~30 j avant expiration). Adoption validée — alternative écartée : certificats manuels, incompatibles avec un flux GitOps automatisé (risque de rupture de disponibilité).
