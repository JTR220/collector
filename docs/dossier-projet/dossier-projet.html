Dossier Projet - collector.shop

Documentation de reference du monorepo : finalite produit, architecture microservices, flux evenementiels, execution locale, infrastructure GitOps et chaine DevSecOps.

Monorepo SvelteKit Go Microservices RabbitMQ Kubernetes GitOps

1. Resume executif

collector.shop est une plateforme orientee collection qui reunit catalogue, marketplace, gamification, historique de prix, detection de fraude, notifications temps reel et messagerie utilisateur. Le projet repose sur un frontend SvelteKit et quatre services Go specialises par domaine.

Composants metier

  • auth-service : identite et JWT
  • catalog-service : articles, ventes, wishlist, avis, moderation
  • price-tracker-service : historique et anti-fraude
  • notification-service : notifications, WebSocket, messages
  • collector-front : experience web unifiee

Capacites techniques

  • communication REST + WebSocket
  • evenements RabbitMQ pour le decouplage
  • Docker Compose en local
  • Kubernetes + Argo CD pour la cible
  • pipelines CI avec scans de securite, SBOM et signatures

5. Flux cles

Connexion

Utilisateur -> Frontend -> POST /login -> auth-service auth-service -> JWT -> Frontend Frontend -> GET /me -> auth-service auth-service -> profil utilisateur -> Frontend

Mise a jour de prix et anti-fraude

Frontend -> PUT /article/:id -> catalog-service catalog-service -> publish price.updated -> RabbitMQ RabbitMQ -> price-tracker-service price-tracker-service -> persist history -> PostgreSQL collector_price price-tracker-service -> detecte spike / flood / dumping price-tracker-service -> publish fraud.alert -> RabbitMQ RabbitMQ -> notification-service notification-service -> persist notification -> PostgreSQL collector_notifications notification-service -> push WebSocket -> navigateur

Achat et notification

Acheteur -> Frontend -> POST /article/:id/buy -> catalog-service catalog-service -> cree la commande catalog-service -> publish order.created -> RabbitMQ RabbitMQ -> notification-service notification-service -> notification vendeur + event temps reel

6. Donnees et conventions

Bases

  • collector : auth et catalogue
  • collector_price : historique et alertes
  • collector_notifications : notifications et messages

Convention d'identifiants

Le catalogue utilise des uint internes, convertis en UUID deterministes pour les services consommateurs et le frontend price tracking / notifications.

8. GitOps et DevSecOps

Commit Git -> GitHub Actions -> lint / tests / SAST / SCA / build image / Trivy -> SBOM + signature cosign -> commit du digest dans overlay staging -> Argo CD synchronise le cluster -> promotion prod par Pull Request puis sync manuel

Le projet integre deja des controles importants : gosec, govulncheck, Semgrep, Trivy, gitleaks, SBOM, cosign et des politiques Kubernetes via Kyverno.

9. Forces et limites

Forces

  • bonne separation des domaines
  • flux event-driven coherents
  • observabilite et supervision presentes
  • chaine CI/CD defendable
  • cible GitOps industrialisable

Limites

  • couverture de tests encore extensible
  • contrats JSON a homogeniser
  • gestion des roles encore simple
  • documentation d'API formelle encore a completer
  • certaines parties restent de niveau POC avance

10. Conclusion

collector.shop constitue un projet complet pour demonstrer une architecture moderne : separation frontend / services, integration asynchrone, stockage segmente, deploiement GitOps et controles DevSecOps integrés.
